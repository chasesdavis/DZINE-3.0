Framework

D-ZINE is a design runtime, not a prompt folder.

The framework parses renderable source, validates design metadata, compiles agent prompts, exports tokens, generates multi-page websites, and audits visual output.

Read docsView proof

Runtime architecture

Every layer has a job, and every output stays traceable to source.

The framework keeps visual intent, machine-readable metadata, generated implementation assets, and QA evidence connected.

Parse

HTML plus JSON

The parser validates doctype, version, root metadata, briefs, DNA, tokens, recipes, rubrics, sections, specimens, and anti-patterns.

Generate

Routes and exports

The site runtime composes page sources into static routes while single-document exporters produce prompt and token artifacts.

Verify

Audit evidence

The audit loop captures responsive screenshots, checks technical quality, and now evaluates taste contracts from the active profile.

Data model

TypeScript schemas make taste enforceable.

D-ZINE uses typed interfaces and schemas for documents, briefs, visual DNA, recipes, token sets, components, anti-patterns, rubrics, reports, and site config.

Document

Root metadata, visible HTML, parsed JSON blocks, diagnostics, and audit-ready structure.

Tokens

DTCG-compatible token trees for color, spacing, radius, and future theme modes.

Site

Project config, routes, source paths, assets, shared theme, navigation, and GitHub repo metadata.

CLI surface

Single document commands and site-level commands work together.

Single-document commands remain useful for focused agent work. Site commands orchestrate the full generated product website.

01

compile

Turns one source file into an agent-ready prompt.

02

export

Writes tokens, CSS variables, Tailwind config, goal prompt, and compatibility Markdown.

03

site build

Generates every configured route into the static website output folder.

04

taste audit

Scores generated source against provenance, proof, typography, rhythm, and anti-slop contracts.

Generated shell

The public site shell is shared, while page bodies stay source-authored.

The generator extracts each page's main content and D-ZINE JSON, then wraps it in a shared header, nav, GitHub star pill, stylesheet, footer, and progressive enhancement script.

Failure modes

The framework is designed to reject shallow design automation.

D-ZINE should fail loudly when an artifact loses structure, accessibility, responsiveness, or design specificity.

Rejected

A design system that proves one impressive page but cannot generate a complete website, docs section, or reusable routes.

Preferred

A project model that scales from one source document to a route tree, docs, generated assets, and route-by-route QA.

Verification

Goal 4 is only complete when the generated site passes.

The framework must build, generate routes, audit each page, and keep the generated website static-host friendly.

Build gate

TypeScript compilation and tests must pass.

Route gate

Home, docs, framework, and showcase routes must be generated.

Audit gate

Each generated route must pass document and visual audit.
