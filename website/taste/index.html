Taste Graph

Taste comes from sources, contracts, critique, and review.

D-ZINE does not ask agents to “make it beautiful” and hope. It loads taste profiles backed by public design docs, canonical systems, product patterns, brand case studies, and local/community learning.

Sources

The Taste Graph starts with inspectable provenance.

Sources include public `DESIGN.md` ecosystems, design token standards, design system galleries, real product pattern libraries, canonical design systems, and brand or agency case studies.

Discover

Source registries

Awesome Design MD Gallery, awesome-design-md, design system galleries, and curated GitHub repos help discover public references.

Distill

Principles, not copies

D-ZINE extracts general rules, constraints, anti-patterns, and rubrics. It does not clone protected visuals or brand identities.

Enforce

Contracts

Profiles reference contracts that can score generated work for specificity, proof, typography, rhythm, and anti-slop behavior.

Profiles

Users can choose or publish taste profiles.

A taste profile is a portable contract: source URLs, license notes, allowed use, principles, examples, anti-patterns, component rules, typography rules, layout rules, copy rules, screenshot notes, and a rubric.

dzine-launch

D-ZINE-specific launch profile for source-backed AI design systems, inspection surfaces, contracts, and open-source proof.

editorial-launch

Expressive launches that need art direction without losing product specificity.

mobile-native

Task-first mobile flows with platform awareness, state clarity, and touch ergonomics.

brand-identity

Identity systems with behavior, misuse rules, applications, typography posture, and voice guidance.

community packs

Future packages like `@dzine/taste-fintech-premium` must include provenance and review status.

Taste audit

The audit checks for generic output that technical audits miss.

A page can pass contrast and responsiveness while still feeling generic. Taste audit adds checks for source declaration, proof density, section rhythm, hype language, fake metrics, and profile completeness.

Command

Run taste audit against a D-ZINE page source and a selected profile.

profilesourcescontractsproofrhythm
node dist/cli/index.js taste list
node dist/cli/index.js taste show dzine-launch
node dist/cli/index.js taste audit dzine/pages/home.dzine.html --profile dzine-launch --out reports/website/taste-home.md

Community

People can bring their own taste, but not without provenance.

D-ZINE should let creators publish templates, profiles, and contracts. The trust model matters: community taste packs are untrusted until reviewed.

01

Create

Author a profile with sources, allowed use, principles, examples, rules, and rubric.

02

Audit

Run it against generated work and prove it catches generic output.

03

Review

Check provenance, licensing, copying risk, examples, and failure modes.

04

Share

Publish as a reusable profile or package after review.

Guardrails

Self-improvement must be reviewed, local, and reversible.

The learning loop stores approved local learnings, but it must not silently mutate global taste profiles or copy designs from references.

Rejected

Scrape inspiration, mutate global rules, copy a reference, and call the output taste.

Preferred

Screenshot output, critique against contracts, ask for approval, store local learning, and propose reviewed community updates.

Proof

Taste now has artifacts in the repo.

The registry, profiles, contracts, critiques, reports, and commands make design judgment inspectable. This is the difference between vibe prompting and a taste system.
