Documentation

Build full websites with D-ZINE, not just single files.

D-ZINE uses `.dzine.html` as the design intelligence layer and a site config as the project layer. Together they generate static routes, shared themes, agent prompts, tokens, screenshots, and audits.

Mental model

D-ZINE separates design intelligence from implementation output.

The source files do not have to be the final app code. They are the visible, parseable design contract that guides agents and verifies results.

Quickstart

Install dependencies, build the TypeScript CLI, generate the website, and audit the generated routes.

pnpm install
pnpm build
node dist/cli/index.js site build --config dzine.config.json --out website
node dist/cli/index.js site audit --config dzine.config.json --out reports/website/site-audit --port 6200

The generated `website/` folder is static HTML and assets. It can be served locally, opened by a static host, or published through GitHub Pages.

The `.dzine.html` format

A D-ZINE page is valid HTML with visible sections plus JSON script blocks. Humans can inspect the rendered page. Agents can parse the metadata. Reviewers can audit screenshots.

  • `dzine-root` identifies the document, modes, source format, quality policy, and exports.
  • `dzine-brief` states the project, audience, objective, constraints, and success criteria.
  • `dzine-dna` defines density, variance, motion, brand posture, copy posture, and banned patterns.
  • `dzine-tokens` carries DTCG-style tokens for colors, radius, spacing, and system roles.
  • `dzine-recipes` names workflows such as web landing, mobile app flow, brand kit, docs, and site generation.
  • `dzine-rubric` defines what quality means before the agent starts designing.

The Taste Graph

D-ZINE does not invent taste from vibes. It loads source-backed taste profiles from `dzine/taste/`, then audits generated work against contracts for provenance, specificity, typography, layout rhythm, product proof, and anti-slop behavior.

dzine/taste/
  registry.json
  sources/
  profiles/
  contracts/
  critiques/
  community/

Seed sources include public `DESIGN.md` ecosystems, design token standards, design system galleries, product pattern libraries, canonical systems, and brand or agency case studies. Sources become structured principles and contracts, not copied visuals.

node dist/cli/index.js taste list
node dist/cli/index.js taste show dzine-launch
node dist/cli/index.js taste audit dzine/pages/home.dzine.html --profile dzine-launch

Project model

A full site is config plus many D-ZINE page sources.

The config controls routes, shared assets, theme, navigation, and GitHub repository metadata. Page sources control the design contract for each route.

`dzine.config.json`

Project manifest for routes, assets, theme, nav items, output directory, and the GitHub stars pill.

`dzine/pages/*.dzine.html`

Page-level design sources. Each route can carry its own brief, DNA, recipes, component specimens, and QA rubric.

`dzine/theme.css`

Shared visual system for generated pages. This keeps public routes cohesive without hiding page-specific design intent.

`website/`

Generated output. This is the folder you link from GitHub, publish with Pages, or serve through any static host.

`reports/website/site-audit`

Route-by-route screenshots and audit reports. The proof travels with the repo.

CLI

The CLI turns source documents into usable product surfaces.

Single-file commands still exist, but Goal 4 adds project-level site commands.

01

`init`

Create a starter D-ZINE project with sample source, tokens, recipes, and docs.

02

`site build`

Generate a multi-page static website from the site config and page sources.

03

`site audit`

Build the site and audit every generated route with responsive screenshots.

04

`export`

Export prompts, goals, design tokens, CSS variables, Tailwind config, and compatibility Markdown.

Agent workflow

How an AI agent should use D-ZINE to build a real product.

The agent reads the source, compiles the prompt, implements the UI, captures screenshots, audits the result, and revises until the evidence is strong.

Agent contract

  1. Read `dzine.config.json` to understand the website shape.
  2. Read the relevant page-level `.dzine.html` source before coding.
  3. Use tokens and recipes as implementation constraints.
  4. Generate or edit output code only after understanding the visible examples.
  5. Run screenshots and audits, then fix failures without lowering the bar.
node dist/cli/index.js compile dzine/pages/home.dzine.html --out reports/home-prompt.txt
node dist/cli/index.js export dzine/pages/home.dzine.html --target all --out reports/home-exports
node dist/cli/index.js site audit --config dzine.config.json --out reports/website/site-audit --port 6200

Quality gates

Anti-slop rules are operational, not inspirational.

D-ZINE bans generic patterns because they create weak products. The audit loop checks layout, contrast, text size, touch targets, section completeness, and reduced motion.

Rejected

Docs that explain the dream but omit commands, file anatomy, route generation, quality gates, and publishing instructions.

Preferred

Docs that teach a builder exactly what files exist, what each command does, what agents should read, and how proof is generated.

Publishing

Generated output is static, repo-friendly, and easy to host.

The `website/` folder contains `index.html`, route folders, `theme.css`, copied assets, and progressive GitHub stars loading. There is no server requirement.

GitHub Pages

Use the included Pages workflow to verify the release and publish the generated `website/` folder through GitHub Actions.

Repo link

The GitHub pill uses the configured owner and repo and fetches live stars from the GitHub REST API.

Fallback

If GitHub is unavailable, the pill remains a normal repository link with readable fallback text.
